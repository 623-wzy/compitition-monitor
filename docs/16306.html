Overview

## 概述 **PCBA标准到真实大挑战赛**关注**跨域视觉问答(VQA)在真实制造检测中的应用**。 印制电路板组装(PCBA)检测不仅需要视觉识别。实际工厂决策取决于**工业标准**和**真实生产线图像**。可靠的检测模型必须理解制造规则、识别上下文相关的缺陷、推理可能的原因，并提供可操作的答案。 本挑战赛评估多模态模型从**标准衍生知识**泛化到**密集、异构真实世界PCBA图像**的能力。邀请参赛者开发VQA系统，能够解读标准文件、识别元器件和缺陷、执行计数任务，并在现实视觉变化下进行推理。 参赛者可在[HuggingFace](https://huggingface.co/datasets/aimmifm/PCBA_Standard-to-Real_Challenge)上申请数据集进行注册，并通过官方竞赛平台提交预测文件。 ## 任务 参赛者获得PCBA相关视觉问题，必须为每个测试样本提交预测答案。基准测试涵盖四个主要VQA类别： 1. **基于标准的知识问答** 源自工业标准的问题，涵盖标准解读、缺陷原因分析和处理决策。 2. **真实性** 感知为中心的问题，包括元器件类型识别、贴装面识别、缺陷存在检测和缺陷类型识别。 3. **定量推理** 数值短答案问题，包括元器件计数和引脚/导线计数。 4. **属性推理** 关于主要元器件形状等视觉属性的问题。 ## 数据集 发布的数据集 https://huggingface.co/datasets/aimmifm/PCBA_Standard-to-Real_Challenge 由两个互补数据源构建： * **标准衍生VQA** 由工业包装标准构建并转换为VQA项目。该子集重点关注标准知识、缺陷原因分析和决策制定。 * **真实世界VQA** 由工厂生产线数据构建。该子集重点关注真实制造图像和实际视觉变化下的检测导向推理。 公开分割包含： | 分割 | 问题数量 | | ---- | -------: | | 训练集 | 8,200 | | 测试集 | 8,200 | | 总计 | 16,400 |

Evaluation

## 评估 所有提交内容都在官方Codabench服务器上自动评估。参赛者必须为发布的公开测试题目提交一个预测文件。公开排行榜报告单一的**总体评分**。 ## 提交格式 对于Codabench提交，参赛者必须上传单个`.zip`文件。`.zip`压缩包必须在根级别包含一个名为`submission.csv`的CSV文件： ```text your_submission.zip └── submission.csv ``` `submission.csv`文件必须包含恰好两列： ```csv qid,answer 1,B 2,12 3,E ``` 提交规则： * `qid`必须与`vqa_test_public.json`中的问题标识符完全匹配 * `answer`必须仅包含最终预测 * 对于多选题，按照问题中显示的选项键提交，例如`A`、`B`、`C`、`D`、`E`或`F` * 对于计数题，以纯字符串形式提交整数，例如`12` * 文件必须为每个测试题目包含一行 * 缺少的行、重复的`qid`值或未知的`qid`值将导致提交无效 * 不要将`submission.csv`放在提交压缩包中的额外子文件夹内 公开测试JSON不公开任务类型注解。任务分组由官方评估服务器在内部处理。 ## 官方指标 每个测试题目在内部属于以下任务组之一： * `standard_knowledge`（标准知识） * `component_type`（元件类型） * `mount_side`（安装侧） * `defect_existence`（缺陷存在性） * `defect_type`（缺陷类型） * `count_component`（元件计数） * `count_pin_lead`（引脚/引线计数） * `attribute_reasoning`（属性推理） 任务级指标： * `standard_knowledge`、`component_type`、`mount_side`、`defect_type`和`attribute_reasoning`采用**准确率** * `defect_existence`采用**F1值** * `count_component`和`count_pin_lead`采用**规范化MAE** 对于`defect_existence`，官方评分使用异常阳性类的二元F1。 对于计数任务，官方规范化评分为： ```text count_score_t = max(0, 1 - MAE_t / C_t) ``` 其中固定的规范化常数为： * `C_count_component = 138` * `C_count_pin_lead = 260` ## 总体评分 每个任务级指标首先转换为相同的`0-1`范围。最终公开排行榜评分随后计算为跨任务组的加权平均值，使用评估集中的任务比例： ```text Overall Score = 100 * sum_t (w_t * s_t) ``` 其中： * `s_t`是任务`t`的任务级评分 * `w_t = n_t / N` * `n_t`是任务`t`的评估题目数量 * `N`是评估题目总数 排行榜显示`0-100`范围内的最终**总体评分**。 ## 评估说明 * 官方Codabench服务器是唯一有效的评分来源 * 组织者侧的任务注解对参赛者隐藏 * 多选题的无效选项标签被计为错误答案 * 计数题的非整数答案被视为无效的计数预测 * 组织者可能在私有评估数据上运行相同的评分协议以进行验证或赛后分析
